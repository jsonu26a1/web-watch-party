there's a small chicken-egg problem; crab_nat needs to know the port number our service
is running on, which it passes on the the NAT device so a mapping can be set up, between
an external port to local port. turn_server must be configured with both a local port to
bind to and an external port/ip. ideally, we would want to bind to any available local
port, give that to crab_nat to setup the mapping, then give the local and external port
to turn_server. but the constructor for turn_server doesn't allow us to do that.

possible ways to handle this:
-   1) ignore it. pick a random port; retry if it fails; let user pick a local port;
    there's a small chance for a collision anyways
-   2) fork turn_server to let us pass an existing socket in; we would worry about
    opening a socket and ensuring the port is available (or letting the OS pick a port)

obviously the answer is 1), but my brain keeps thinking about 2). I wonder if there's a
reason the API doesn't just allow passing in an already created socket?
`turn_server::startup` is the only entry point that I can tell.
